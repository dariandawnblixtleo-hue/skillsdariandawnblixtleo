# Upgrade procedure — `web3-dev` target

Refresh the `web3-dev` skill's bundled OpenAPI specification and endpoint index from the latest commit of the upstream `pro-api` repository. The two artefacts the skill ships with are:

- `web3-dev/references/pro-api.json` — full Blockscout PRO API OpenAPI v3.0 document (~24,000 lines).
- `web3-dev/references/pro-api-index.md` — generated by-tag endpoint index derived from the JSON above.

The OpenAPI document itself is built from a build-time clone of the `pro-api` repository, not authored in this repo. The pipeline is therefore: clone the pinned commit into the build directory → rebuild the OpenAPI doc inside it → copy the doc into the skill → regenerate the index.

## Prerequisites

- Python >= 3.9 (standard library only — the indexer has no third-party dependencies)
- `bash`, `git`, `jq`, and `oastools` available on `PATH` (used by the upstream `build-pro-api.sh` orchestrator and its child scripts)
- Network access to GitHub for the `pro-api` clone and to whatever sources `build-pro-api.sh` downloads (it caches commit-hash-keyed swagger files)
- Working directory is the repository root (parent of `web3-dev/` and `.memory_bank/`)
- Read access to the private `https://github.com/blockscout/pro-api` repository

## Pipeline

Run every step in order. Each step depends on the output of the previous one. If a step fails, fix the cause before proceeding — do not skip ahead.

### Step 1 — Clone the `pro-api` repo into the build directory

The OpenAPI source is a build-time clone of `https://github.com/blockscout/pro-api` into `web3-dev/.build/pro-api`. The build directory is gitignored (`web3-dev/.gitignore`); nothing under `.build/` is ever committed.

The pinned commit is:

```
1e841564d51621a451f57d22982b2ac4d82c44ef
```

> **Temporary pin.** This commit lives on the unmerged branch `feat/openapi-spec-intro`. As soon as [blockscout/pro-api#23](https://github.com/blockscout/pro-api/pull/23) merges to `main`, replace this section with a plain shallow clone of `main` (e.g. `git clone --depth 1 https://github.com/blockscout/pro-api web3-dev/.build/pro-api`) and delete the SHA-pin wording.

Fetch only the pinned commit, with no other history:

```bash
rm -rf web3-dev/.build/pro-api
mkdir -p web3-dev/.build/pro-api
git -C web3-dev/.build/pro-api init -q
git -C web3-dev/.build/pro-api remote add origin https://github.com/blockscout/pro-api
git -C web3-dev/.build/pro-api fetch --depth 1 origin 1e841564d51621a451f57d22982b2ac4d82c44ef
git -C web3-dev/.build/pro-api checkout -q FETCH_HEAD
```

GitHub supports fetch-by-SHA for repos with the default `uploadpack.allowReachableSHA1InWant` setting; this should succeed without listing branches.

Confirm the clone landed and the build script is in place:

```bash
test -x web3-dev/.build/pro-api/openapi-spec/build-pro-api.sh
```

### Step 2 — Rebuild `pro-api.json`

Run the upstream orchestrator that downloads the source swaggers, runs per-service extractions, and merges them into a single OpenAPI 3.0 document. The script is invoked from inside the cloned directory.

```bash
bash web3-dev/.build/pro-api/openapi-spec/build-pro-api.sh
```

The script honours its own caching layer keyed by upstream commit hashes. Use `--force` if you need to re-download even when the cache is warm:

```bash
bash web3-dev/.build/pro-api/openapi-spec/build-pro-api.sh --force
```

On success the script writes:

```
web3-dev/.build/pro-api/openapi-spec/pro-api.json
```

Verify the file exists and is non-trivial in size before moving on:

```bash
ls -l web3-dev/.build/pro-api/openapi-spec/pro-api.json
```

The script's exit codes are documented at the top of `build-pro-api.sh`: `0` success, `1` config error, `2` download failure, `3` extraction failure, `4` merge/validation failure. Any non-zero exit means the artefact is unusable — do not copy it forward.

### Step 3 — Copy the OpenAPI doc into the skill

Overwrite the bundled copy with the freshly built one:

```bash
cp web3-dev/.build/pro-api/openapi-spec/pro-api.json \
   web3-dev/references/pro-api.json
```

This is a single file replacement — no merge, no patch. The bundled file is the authoritative copy from the agent's point of view; the file under `.build/` only exists as a build-time intermediate.

### Step 4 — Regenerate the endpoint index

```bash
python3 .memory_bank/specs/web3-dev/tools/pro-api-indexer.py \
    web3-dev/references/pro-api.json
```

The input path is **mandatory** — the script has no default. Always pass the bundled `web3-dev/references/pro-api.json` (the freshly copied file from Step 3), not the build-directory copy. The script **overwrites** `web3-dev/references/pro-api-index.md`. Output order is deterministic (tag sections sorted case-insensitively; entries within a section sorted by path then HTTP method), so a clean rerun on the same JSON produces a byte-identical index.

The indexer's exit codes: `0` success, `1` input file not found / unreadable, `2` input not valid JSON, `3` JSON has no `paths` key. Any non-zero exit means the index was not written.

## Verification

After completing all four steps, confirm:

1. `web3-dev/references/pro-api.json` is well-formed JSON containing a `paths` object:
   ```bash
   jq '.paths | length' web3-dev/references/pro-api.json
   ```
   should print a positive integer (number of operation paths).
2. `web3-dev/references/pro-api-index.md` exists, starts with `# PRO API Endpoint Index`, and has one `## <tag>` section per OpenAPI tag in the spec.
3. `git diff web3-dev/references/pro-api.json web3-dev/references/pro-api-index.md` shows additions/removals consistent with the spec changes (new endpoints appear under their tags; renamed/removed ones disappear). If the diff is empty after a non-trivial commit bump, something silently failed in Step 2 or 3.
4. Re-running Step 4 against the same `pro-api.json` produces no further changes (the indexer is deterministic).
